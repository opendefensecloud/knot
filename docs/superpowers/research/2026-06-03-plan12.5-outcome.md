## Architecture summary

**toxiproxy** (Shopify) is a TCP proxy with an HTTP admin API for injecting failure modes. We run it as a sidecar in `deploy/compose/dev.yml` with `network_mode: host` so it can reach the host's knot-server. The proxy listener is `:3001`, upstream is `127.0.0.1:3000`, admin is `:8474`. The proxy definition is loaded at startup via `deploy/compose/toxiproxy/init.json` — no separate registration step.

**`VITE_COLLAB_VIA_PROXY=1`** routes the Vite dev server's `/collab` WS proxy target through `ws://localhost:3001` instead of `:3000`. Set unconditionally by the Playwright config's web webServer env, so every e2e test goes through toxiproxy. With no toxics applied the latency cost is negligible (sub-ms forwarding); functionality is identical.

**The reconnect spec** uses toxiproxy's `POST /proxies/knot {"enabled": false}` to forcibly close all live connections — including the active WebSocket — and reject new ones. Re-enabling reopens the listener; KnotProvider's `scheduleReconnect` (Plan 6) backoff kicks in and reconnects on the next attempt. Y.Doc state survives because Yjs holds the document in client memory, independent of WS lifecycle.

```
browser    Vite (:5173)      toxiproxy (:3001)    knot-server (:3000)
   │            │                    │                    │
   │ /collab    │                    │                    │
   │──── WS ───►├── proxy ws ───────►├── proxy tcp ──────►│
   │            │                    │                    │
   │            │ POST /proxies/knot enabled=false        │
   │            │                    │ X kill all sockets │
   │ onclose ◄──┤                    │                    │
   │            │                    │                    │
   │            │ POST /proxies/knot enabled=true         │
   │            │                    │ → accept           │
   │ retry ────►│ → ───────────────►│ → ─────────────────►│
   │ onopen ◄───┤                    │                    │
```

## What was non-obvious

**`reset_peer` doesn't kill existing sockets.** First attempt used the `reset_peer` toxic with `stream: "upstream"`. It applied without error but the WS stayed connected — `reset_peer` only injects RSTs on NEW connections. Fixed by switching to the proxy-level `enabled: false` toggle, which is the documented "close everything" knob.

**Vite caches its proxy config across `pnpm dev` runs.** When you add a new env-driven proxy target, kill the existing Vite process or it'll keep routing to the old target. Playwright config has `reuseExistingServer: !process.env.CI` so dev iterations reuse a long-running Vite — a one-time `pkill -f vite` is the workaround during development. CI doesn't have this issue.

**`network_mode: host` is Linux-only.** On macOS the equivalent is `host.docker.internal` as the upstream host and a published port (`ports: ["3001:3001", "8474:8474"]`). Documented inline. The dev-compose target audience is Linux + Nix flake users so we kept the host network mode for simplicity.

**`[vite] ws proxy socket error: ECONNRESET` log noise.** Expected — that's Vite noticing the upstream WS got force-killed. Could be silenced with a custom logger, but it's a useful signal in dev so we leave it.
